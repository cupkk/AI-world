# AI-World 前后端字段兼容映射（Frontend Adapter Contract）

更新时间：2026-03-01

用途：本页用于后端会话直接对齐当前前端适配层可兼容的字段与响应结构，避免联调时因命名差异阻塞。

---

## 0. 全局约定

- 基础 URL：`VITE_API_BASE_URL`（可空，空时使用相对路径）
- 响应包裹兼容：前端统一支持
  - 直接对象/数组
  - `{ data: ... }`
- 列表结构兼容：优先直接数组，其次 `{ items: [...] }`（部分接口还兼容 `{ queue: [...] }`）

---

## 1. Hub / Talent

### 1.1 `GET /api/hub`

请求 query：
- `type`（可选）
- `q`（可选）
- `tags`（可选，逗号拼接）

响应兼容：
- `Content[]`
- `{ data: Content[] }`
- `{ items: Content[] }`
- `{ data: { items: Content[] } }`

`Content` 字段（前端标准）：
- `id`, `title`, `description`, `type`, `status`, `authorId`, `createdAt`, `tags`, `likes`, `views`, `coverImage?`, `visibility?`, `rejectReason?`

说明：
- `tags` 非数组时会被兜底为空数组。
- `likes/views` 非数字会转 `0`。

### 1.2 `GET /api/talent`

请求 query：
- `q`（可选）
- `tags`（可选，逗号拼接）

响应兼容：
- `User[]`
- `{ data: User[] }`
- `{ items: User[] }`
- `{ data: { items: User[] } }`

`User` 字段映射（兼容）：
- `avatar` <- `avatar | avatarUrl`
- `role` 仅接受：`EXPERT | LEARNER | ENTERPRISE_LEADER | ADMIN`，其它值前端降级为 `LEARNER`
- `skills/blockedUsers` 仅保留字符串数组元素

## 2. Auth / Invite

### 2.1 `POST /api/auth/login`

请求：
- `{ email, password }`

响应兼容：
- `User`
- `{ user: User }`
- `{ data: User }`
- `{ data: { user: User } }`

### 2.2 `POST /api/auth/invite/verify`

请求：
- `{ code }`

响应兼容：
- `InviteCode`
- `{ invite: InviteCode }`
- `{ valid: false }`（表示无效）
- 上述结构再套 `{ data: ... }`

`InviteCode` 字段：
- `id`（缺失时前端补 `inv-${timestamp}`）
- `code`（缺失时回退请求 code）
- `status` 仅接受 `UNUSED | USED | REVOKED`，其它值降级为 `UNUSED`
- `issuedByAdminId?`, `boundUserId?`, `expiresAt?`, `createdAt`

## 3. Publish / Review

### 3.1 `POST /api/publish`

请求：
- `{ title, description, type, tags, visibility? }`

响应兼容：
- `Content`
- `{ item: Content }`
- `{ content: Content }`
- 上述结构再套 `{ data: ... }`

### 3.2 `POST /api/publish/:id/submit`

响应兼容：
- `Content`
- `{ item: Content }`
- `{ content: Content }`
- `null`（前端可接受）
- 上述结构再套 `{ data: ... }`

### 3.3 `GET /api/publish/mine`

响应兼容：
- `Content[]`
- `{ items: Content[] }`
- 上述结构再套 `{ data: ... }`

### 3.4 `GET /api/admin/review`

响应兼容：
- `Content[]`
- `{ items: Content[] }`
- `{ queue: Content[] }`
- 上述结构再套 `{ data: ... }`

### 3.5 `POST /api/admin/review/:id/approve`
### 3.6 `POST /api/admin/review/:id/reject`

请求（reject）：
- `{ reason }`

响应：
- 仅校验 HTTP 2xx；不依赖响应体

## 4. Messages REST

### 4.1 `GET /api/messages/conversations`
### 4.2 `GET /api/messages/requests`

响应兼容：
- `ChatThread[]`
- `{ items: ChatThread[] }`
- 上述结构再套 `{ data: ... }`

`ChatThread` 字段映射：
- `id` <- `id | conversationId`
- `participants` 支持：
  - 用户对象数组
  - 用户 id 数组（会按本地用户表补全）
  - 或 `participantIds`
- `lastMessage`（可选）会走消息映射
- `status` 仅接受：`PENDING | ACCEPTED | REJECTED`，其它值降级为 `PENDING`
- `initiatorId` <- `initiatorId | fromUserId`

### 4.3 `POST /api/messages/conversations`

请求：
- `{ targetUserId }`

响应兼容：
- `ChatThread`
- `{ item: ChatThread }`
- `{ conversation: ChatThread }`
- 上述结构再套 `{ data: ... }`

### 4.4 `GET /api/messages/conversations/:cid/messages`

响应兼容：
- `Message[]`
- `{ items: Message[] }`
- 上述结构再套 `{ data: ... }`

### 4.5 `POST /api/messages/conversations/:cid/messages`

请求：
- `{ bodyText }`

响应兼容：
- `Message`
- `{ item: Message }`
- `{ message: Message }`
- `null`
- 上述结构再套 `{ data: ... }`

`Message` 字段映射：
- `id` <- `id`
- `senderId` <- `senderId | fromUserId`
- `receiverId` <- `receiverId | toUserId`
- `content` <- `content | bodyText`
- `createdAt` <- `createdAt`
- `read` <- `read`（缺失默认 `false`）

### 4.6 `POST /api/messages/conversations/:cid/read`
### 4.7 `POST /api/messages/requests/:id/accept`
### 4.8 `POST /api/messages/requests/:id/reject`

响应：
- 仅校验 HTTP 2xx；不依赖响应体

## 5. WebSocket 事件（/ws）

### 5.1 `message:new`

前端兼容字段：
- 会话 id：`conversationId | threadId | conversation.id`
- 消息 id：`id`
- 发送方：`senderId | fromUserId`
- 接收方：`receiverId | toUserId`
- 内容：`content | bodyText`
- 时间：`createdAt`
- 已读：`read`

### 5.2 `conversation:update`

前端兼容字段：
- 会话 id：`conversationId | id`
- 未读数：`unreadCount`（number）
- 状态：`PENDING | ACCEPTED | REJECTED`

## 6. Assistant / Knowledge Base

### 6.1 `POST /api/assistant/recommend`

请求：
- `{ query, userId?, locale?, history? }`

响应兼容：
- `reply` <- `reply | message | content | text`
- `recommendedUserId` <- `recommendedUserId | userId | recommended_user_id | recommendations[type=user].id`
- `recommendedContentId` <- `recommendedContentId | contentId | recommended_content_id | recommendations[type=content].id`
- 支持响应外层 `{ data: ... }`

### 6.2 `POST /api/knowledge-base/upload`

请求：
- `multipart/form-data`，字段名：`file`

响应兼容：
- `id` <- `id | fileId`
- `name` <- `name | fileName`
- `type` <- `type | mimeType`
- `status` 仅接受：`PROCESSING | READY | FAILED`
- `uploadedAt` <- `uploadedAt | createdAt`
- 支持响应外层 `{ data: ... }`

## 7. 后端对齐建议（最小集合）

建议后端优先稳定以下“首选命名”，可减少前端兼容分支：

- 统一列表：`{ items: [...] }`
- 统一对象：`{ data: {...} }`
- 用户头像字段统一用 `avatar`
- 消息内容字段统一用 `content`（REST 与 WS 一致）
- 会话主键统一用 `conversationId` 或 `id` 二选一（建议 `id`）
- 审核队列统一返回 `items`
