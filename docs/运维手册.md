# AI-World 运维手册

更新时间：2026-03-12

## 1. 架构概览

当前部署模型为单台 ECS + Docker Compose 双栈：

- edge: `server/ops/docker-compose.edge.yml`
- production: `server/docker-compose.prod.yml`
- staging: `server/docker-compose.staging.yml`
- observability: `server/ops/docker-compose.observability.yml`

域名规划：

- production
  - `https://ai-world.asia`
  - `https://www.ai-world.asia`
- staging
  - `https://staging.ai-world.asia`

公网只暴露：

- `GET /ready`
- `GET /health`

`/metrics` 只允许通过 Docker 内网抓取：

- `production-api:3000/metrics`
- `staging-api:3000/metrics`

## 2. 服务器目录

服务器部署目录约定：

```bash
/opt/aiworld
```

常用工作目录：

```bash
cd /opt/aiworld/server
```

备份目录约定：

```bash
/opt/aiworld/backups/production
/opt/aiworld/backups/staging
```

## 3. 外部前置项

上线前必须确认以下资源已经就绪：

1. ECS 已绑定固定 EIP
2. DNS A 记录已生效
   - `@ -> <EIP>`
   - `www -> <EIP>`
   - `staging -> <EIP>`
3. HTTPS 证书覆盖
   - `ai-world.asia`
   - `www.ai-world.asia`
   - `staging.ai-world.asia`
4. 生产与 staging 的 SMTP / OSS / LLM 凭据已填写
5. GitHub Actions secrets 已填写
   - 参见 [GitHub_Actions_Secrets.md](/d:/github/AI-world/docs/GitHub_Actions_Secrets.md)

## 4. 环境文件

首次部署前复制模板：

```bash
cd /opt/aiworld/server
cp .env.production.example .env.production
cp .env.staging.example .env.staging
```

生产和 staging 必须满足：

- `REQUIRE_OSS=true`
- `REQUIRE_SMTP=true`
- `REQUIRE_LLM=true`
- `ENABLE_DEMO_INVITES=false`
- `SEED_INCLUDE_DEMO_DATA=false`

前端构建必须关闭 demo：

- `VITE_ENABLE_DEMO_MODE=false`
- `VITE_ALLOW_DEMO_PREFILL=false`

## 5. 网络与基础容器

首次部署执行：

```bash
cd /opt/aiworld/server
chmod +x ./scripts/*.sh
./scripts/ensure-docker-networks.sh
```

会创建：

- `aiworld-edge`
- `aiworld-observability`

启动 edge：

```bash
cd /opt/aiworld/server
docker compose -f ./ops/docker-compose.edge.yml up -d
```

启动 observability：

```bash
cd /opt/aiworld/server
docker compose -f ./ops/docker-compose.observability.yml up -d
```

校验监控抓取：

```bash
cd /opt/aiworld/server
./scripts/verify-observability.sh
```

## 6. 证书

扩签或更新证书：

```bash
cd /opt/aiworld/server
./scripts/issue-certificate.sh ai-world.asia www.ai-world.asia staging.ai-world.asia
```

先做 dry-run：

```bash
cd /opt/aiworld/server

```

## 7. 部署与回滚

### 7.1 部署 staging

```bash
cd /opt/aiworld/server
REGISTRY=<registry> IMAGE_TAG=<tag> ./scripts/deploy-stack.sh staging
./scripts/check-stack.sh staging
```

### 7.2 部署 production

```bash
cd /opt/aiworld/server
REGISTRY=<registry> IMAGE_TAG=<tag> ./scripts/deploy-stack.sh production
./scripts/check-stack.sh production
```

### 7.3 关闭 staging

```bash
cd /opt/aiworld/server
docker compose -p aiworld-staging -f ./docker-compose.staging.yml down
```

### 7.4 回滚 production

```bash
cd /opt/aiworld/server
REGISTRY=<registry> ./scripts/rollback-stack.sh production <previous-tag>
./scripts/check-stack.sh production
```

回滚后必须重新跑 production smoke。

## 8. GitHub Actions 发布链路

工作流：

- `ci.yml`
- `deploy-staging.yml`
- `promote-production.yml`

要求：

1. `main -> staging` 自动部署
2. staging live smoke 通过后，才允许 production promote
3. production 只允许提升已通过 staging 的镜像 tag
4. 不允许在生产机临时重新 `docker build`

## 11. 常见排障

### 11.1 Production 或 staging API 不健康

```bash
cd /opt/aiworld/server
docker compose -p aiworld-production -f ./docker-compose.prod.yml ps
docker compose -p aiworld-production -f ./docker-compose.prod.yml logs api --tail 200
docker compose -p aiworld-staging -f ./docker-compose.staging.yml ps
docker compose -p aiworld-staging -f ./docker-compose.staging.yml logs api --tail 200
```

### 11.2 Staging 域名证书错误

优先检查：

1. `staging.ai-world.asia` 是否已解析到固定 EIP
2. edge 是否运行
3. 证书是否已扩到 `staging.ai-world.asia`
4. `./scripts/issue-certificate.sh --dry-run ...` 是否成功

### 11.3 `/metrics` 公网访问失败

这是预期行为。公网应返回 `404`。

内部检查：

```bash
cd /opt/aiworld/server
./scripts/verify-observability.sh
```

### 11.4 AI 助手返回 503

检查：

1. `REQUIRE_LLM=true`
2. `LLM_API_KEY` 是否已配置
3. 上游模型服务是否可用

错误响应：

```json
{
  "code": 503,
  "errorCode": "ASSISTANT_UNAVAILABLE",
  "message": "..."
}
```
