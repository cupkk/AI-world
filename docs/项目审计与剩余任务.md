# 项目审计与剩余任务

更新时间：2026-03-15

## 1. 当前项目状态

AI-World 已经不是“原型站”或“静态演示页集合”，而是一个具备持续开发、持续部署和线上验收能力的产品仓库。

- 前端：Vite + React 19 + React Router + Zustand
- 后端：NestJS + Prisma + PostgreSQL + Redis + Socket.IO
- 异步处理：Python worker，负责知识库文件解析、切块、向量化
- 部署：单机 ECS 同时承载 `staging / production / observability`
- 发布链路：GitHub Actions 已打通 `CI -> Deploy Staging -> Promote Production`
- 线上验收：`staging smoke / staging mutation / promote production` 已跑通

从工程角度看，项目已经具备：

- 可开发
- 可测试
- 可部署
- 可回归
- 可在 staging 先验证，再提升到 production

当前真正的短板已经不再是“有没有功能”，而是：

- 领域边界是否足够清晰
- 读模型是否统一
- 跨模块回归是否足够厚
- AI / 知识库链路是否足够稳定

## 4. 当前最明显的不足

### 4.1 内容域第二层边界还没收干净

虽然 `publish / hub` 第一层边界已经清理，并且 `HubDetail / Profile` 两个页面都已切到聚合读模型，但内容域第二层边界还没彻底收干净：

- 通用 Hub 内容、企业需求、专家研究项目三条内容线的详情语义仍未完全统一
- 页面聚合接口已经有了，但跨内容类型的公共字段和专属字段边界仍需要继续收口

### 4.2 三类内容语义仍不够统一

项目里现在有三条内容线：

- 通用 Hub 内容
- 企业需求
- 专家研究项目

它们已经能复用部分发布与申请能力，但接口语义、详情读模型、页面职责仍未完全统一。

### 4.3 申请系统虽然有统一读接口，但 audit 侧还没真正用起来

`outbox / inbox / audit` 第一版已经完成，但当前主要落地的是：

- 企业 `inbox`
- 专家 `inbox`
- 学员 `outbox`

还没有一套明确的“管理员申请审计 / 异常处理”页面或聚合入口真正消费 `audit`。

### 4.4 跨模块流程回归已有主链，但还没扩成全覆盖

这一轮已经补上两条最关键的端到端业务链路：

- 发布 -> 审核 -> HubDetail 展示
- 申请 -> 拥有者处理 -> 工作台状态变化

剩余缺口主要变成“变体覆盖”而不是“主链缺失”，例如：

- reject / resubmit / re-approve 的多次流转
- enterprise / expert 两条 owner 处理线的更细分支
- 资料更新 -> Profile 展示一致性

### 4.5 AI / 知识库仍属于“可用版”，不是“稳定版”

- Worker 虽可运行，但缺少更完整的失败原因、重放、时间线与观测信息
- Assistant 缺会话持久化、引用来源、反馈闭环
- 当前更像可运营 MVP，而不是长期稳定的 AI 基础能力

## 5. 剩余任务分级

## P0：必须继续完成

- 做内容域第二层语义收口
  - 明确通用内容、企业需求、研究项目三者的公共字段与专属字段
  - 避免继续出现三套详情页语义各自演化

- 把申请系统 `audit` 真正落到可见页面或管理入口
  - 管理员可查看申请审计视图
  - 支持异常状态排查与后续治理

- 扩展跨模块流程回归的变体覆盖
  - reject / resubmit / re-approve
  - enterprise / expert owner 双线处理
  - 资料更新 -> Profile 展示

## P1：强烈建议尽快完成

- 增强知识库处理链路的可观测性、失败原因与重放能力
- 增强 Assistant 的会话持久化、引用来源与反馈闭环
- 继续统一企业需求 / 研究项目 / 通用内容三条线的接口语义
- 强化管理员后台运营能力
  - 举报治理
  - 用户治理
  - 审计日志查询
  - 更细粒度筛选与批量操作

## P2：下一阶段产品迭代

- 更完整的对外品牌首页 / 落地页
- 更强的通知系统与运营提醒
- 更成熟的 RAG / 知识库驱动 Assistant
- 数据看板与运营分析

## 6. 建议的执行顺序

按当前项目状态，建议继续按下面顺序推进：

1. 先把 `audit` 读模型落到管理员视图
2. 再继续做内容域第二层语义收口
3. 接着扩展跨模块流程回归的变体覆盖
4. 最后推进 AI / 知识库链路的稳定性和可观测性

## 7. 当前结论

AI-World 现在已经是“可以持续演进的产品仓库”，不是演示项目，也不是只会堆页面的壳子。

当前已经完成的阶段性目标是：

- 线上发布链路打通
- 角色工作台聚合接口化
- 管理员内容管理聚合化
- `publish / hub` 第一层边界清理
- 申请系统统一读接口第一版
- 企业 / 专家 / 学员对统一申请读模型的第一轮接入
- 单测与 E2E 已补一轮

接下来真正的主线，不再是“补几个页面”，而是：

- 把内容域第二层边界做干净
- 把详情页与作者页变成真正的聚合读模型
- 把申请系统做成完整、稳定、可治理的角色化读模型
- 把跨模块回归和 AI / 知识库链路做厚
